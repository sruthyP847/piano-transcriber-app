## 1. Project Overview

**What this app does:** takes a bird's-eye-view video of someone playing piano and automatically generates professional, print-ready sheet music (a PDF) from it. Upload a video → the backend extracts audio, detects notes, works out rhythm/bar structure, and renders real engraved music notation.

**Architecture — two tiers:**
- **Backend**: FastAPI, single file at `backend/main.py`. Does all the real work: audio extraction (moviepy), pitch/note detection (basic-pitch neural model), note deduplication and event grouping, confidence-based filtering, bar/beat/rhythm computation, and PDF notation rendering (abjad + LilyPond).
- **Frontend**: Next.js/React, single file at `frontend/app/page.tsx`. A drag-and-drop upload UI that, on success, shows a split-screen dashboard: video player, an "Audio Properties" card (tempo/duration/sample rate), a "Detected Events" card, a "Raw Notes" card (both are debug-oriented JSON-derived lists, not a polished end-user view), download buttons for the PDF/MusicXML, and an embedded PDF preview via `<iframe>`.

**The original four-phase blueprint** (pitch detection → temporal grouping → music theory engine → frontend timeline) has been **substantially superseded** by what was actually built. Specifics:

- **Phase 1 (pitch detection)**: originally a hand-rolled CQT (Constant-Q Transform) + peak-picking approach, built and tested across several early sessions. This was **fully replaced** by a pretrained neural model (`basic-pitch`, Spotify) after a head-to-head investigation against both the CQT approach and a second candidate model (ByteDance's `piano_transcription_inference`). See Section 3 for why.
- **Phase 2 (temporal grouping)**: exists, but evolved substantially — onset grouping, confidence filtering, and decay-tail suppression were each built, tested against real audio, found to have real bugs, and iteratively fixed (see Section 3 and Section 4 for the specific history). This phase is **not a single clean implementation** — it's the result of several rounds of "implement → test against real data → find a real bug → fix" cycles, and that iterative history matters for anyone touching this code.
- **Phase 3 (music theory engine — chord naming, key detection)**: **not built at all**. The pipeline currently outputs raw note/chord content and rhythm, with zero harmonic analysis layer. This is a real, acknowledged gap (Section 4).
- **Phase 4 (frontend timeline)**: **not built as originally conceived**. What exists instead is a simple two-column dashboard with plain data-dump cards ("Raw Notes", "Detected Events") and an embedded PDF preview — there is no interactive timeline UI showing notes against a playhead or similar. The actual "phase 4" deliverable that DID get built, and that wasn't in the original four-phase framing at all, is the **notation rendering pipeline** (abjad/LilyPond, Stages A/B/B2/C across several sessions) — turning detected notes into an actual engraved PDF score. This became the real focus of the project's second half and is now the most mature, heavily-validated part of the codebase.

## 2. Current Pipeline Architecture (verified against actual code in `backend/main.py`, commit `5ee7ad1`)

### `/api/transcribe` endpoint, step by step:

1. **Upload validation** — content-type must be one of `video/mp4`, `video/quicktime`, `video/x-m4v` (`ALLOWED_CONTENT_TYPES`).
2. **Save the uploaded video** to `backend/uploads/<uuid>.<ext>`.
3. **Audio extraction** — `VideoFileClip(...).audio.write_audiofile(...)` (moviepy), saved as `<uuid>.wav`, 44.1kHz 16-bit PCM.
4. **`analyze_audio(audio_path, video_path)`** does essentially the entire analysis pipeline:
   a. Loads the waveform via `librosa.load(sr=None)` (native sample rate preserved).
   b. Computes `duration_seconds` and a single global `tempo_bpm` via `librosa.beat.beat_track`.
   c. **Note detection**: `detect_notes_basic_pitch(audio_path)` — runs `basic_pitch.inference.predict()` with the `ICASSP_2022_MODEL_PATH` model, returns a flat list of raw note dicts (`onset`, `offset`, `note` [via `librosa.midi_to_note`, unicode ♯/♭], `midi`, `confidence`), sorted by onset. This is `raw_notes` in the API response — completely unfiltered.
   d. **Deduplication**: `deduplicate_notes(notes)` — groups raw notes by MIDI pitch, then merges same-pitch fragments that are within `DEDUP_GAP_SECONDS` (0.05s) of each other AND whose combined span doesn't exceed `DEDUP_MAX_SPAN_SECONDS` (2.0s). Fixes basic-pitch's tendency to report one held note as 2+ consecutive fragments. The span cap exists specifically because the naive version of this fix caused a real regression — see Section 3/6.
   e. **Event grouping**: `group_notes_into_events(deduped)` — chains notes into "events" (chords/rolls/single notes) using chained-distance grouping within `GROUPING_WINDOW_SECONDS` (0.15s), classifying each event's `style` as `"single"` (1 note), `"solid"` (multiple notes, span ≤ `SOLID_THRESHOLD_SECONDS`=0.03s — a tightly-struck chord), or `"rolled"` (spread arpeggio-like attack).
   f. **Confidence filtering**: `filter_event_notes(events)` — **per-event, group-relative** filtering. A note survives only if its confidence clears BOTH an absolute floor (`ABSOLUTE_CONFIDENCE_FLOOR`=0.35) AND a fraction of its own event's strongest note (`RELATIVE_CONFIDENCE_FRACTION`=0.6). This is NOT a single global threshold — see Section 3 for why that was tried and rejected.
   g. **Decay-tail suppression**: `suppress_decay_tail_notes(filtered_events)` — drops short notes (< `DECAY_TAIL_DURATION_THRESHOLD`=0.4s) whose pitch class matches a note in one of the previous `DECAY_TAIL_PITCH_LOOKBACK_EVENTS` (2) events within `DECAY_TAIL_DECAY_MARGIN_SECONDS` (0.5s) — targets the case where a string's decaying resonance gets mistaken by basic-pitch for a fresh onset.
   h. **Bar/beat/rhythm computation** (reintegrated in Stage C after being dead code since the basic-pitch migration): `event_times` extracted from the FINAL filtered/suppressed events → `convert_seconds_to_beats(event_times, tempo_bpm)` → `quantize_beats(...)` (snaps to nearest 0.25-beat / sixteenth-note grid) → `calculate_bar_structures(...)` (1-indexed bar number + beat-in-bar position) → `calculate_note_durations(...)` (duration = **gap to the next quantized event's beat position**, NOT basic-pitch's acoustic offset time — see Section 3 for why this distinction matters). Results (`bar`, `beat_in_bar`, `duration_beats`, `note_type`) are attached directly onto each event dict.
   i. A diagnostic frame-check loop (`extract_frame_at_time`) runs against the first 3 note onsets, printing whether a video frame could be read at that timestamp. **This is a sanity-check leftover from early multimodal-architecture planning, not a real feature** — it does not cross-reference detected notes against depressed keys in any way. See Section 4.
   j. Returns a dict: `duration_seconds`, `sample_rate`, `tempo_bpm`, `raw_notes` (unfiltered), `events` (final, filtered, with bar/beat/duration/note_type attached).
5. **Notation rendering**: `generate_notation_pdf(events, pdf_path)` — builds a two-staff grand staff (treble ≥ C4, bass < C4) via `abjad`, using `spell_rhythm()` for beat-respecting rhythm notation (see Section 3), and renders to PDF via LilyPond (`abjad.persist.as_pdf`). This is a **real, permanent function** now — no longer a placeholder.
6. **MusicXML**: `generate_placeholder_musicxml(musicxml_path)` — still writes a **static, hardcoded single-measure placeholder XML file**, completely disconnected from the real detected notes. This is deliberate, not an oversight — see Section 3.
7. Response JSON includes `pdf_url` and `musicxml_url` (both served via `/api/uploads/...` static mount) plus everything from `analyze_audio`'s return dict.

### Notation rendering internals (`generate_notation_pdf` and helpers):
- `spell_rhythm(start, duration, beats_per_bar=4, sixteenths_per_beat=4)` — a recursive, beat-respecting rhythm-spelling algorithm operating in integer sixteenth-note units. Handles single AND double dots. Three-tier priority: (1) never let a value cross a bar line, (2) a value starting exactly on a beat boundary can be notated as one symbol (this deliberately includes dotted/double-dotted values even when they extend past a subsequent beat boundary — see Section 3), (3) within-beat greedy largest-fit for anything else, with ties for remainders.
- `note_name_to_lilypond_pitch` / `note_name_to_midi` — parse note names in the exact format `librosa.midi_to_note()` produces (unicode `♯`/`♭`, e.g. `"C♯4"`), and convert to abjad's pitch string format. **Important gotcha already found and fixed**: `abjad.Staff()` defaults to `language="english"`, which uses plain `s`/`f` suffixes for sharp/flat (`cs`=C♯, `ef`=E♭) — NOT the Dutch-style `is`/`es` convention. This bug existed through two full validation stages (A and B) because neither hardcoded test dataset happened to include an accidental, and only surfaced when real detected data (which has plenty of accidentals) was wired in during Stage C.
- `_build_staff_input` — walks events in order, splits each event's notes by pitch (≥C4 treble, else bass) — multiple notes on one staff render as a genuine LilyPond chord (`<c' e' g'>`), fills gaps with rests via `spell_rhythm`, appends a trailing rest to round out the final bar.

## 3. Key Architectural Decisions and WHY

**basic-pitch chosen over hand-rolled CQT and over `piano_transcription_inference`.** A three-way investigation was run: the original hand-rolled CQT+peak-picking approach, Spotify's `basic-pitch`, and ByteDance's `piano_transcription_inference`. Findings that decided it:
- CQT approach: pervasive octave-harmonic phantom notes (a struck note's overtone at +12/+19 semitones frequently detected as a separate "played" note), and a catastrophic failure mode where a near-silent tail produced an 8-note noise-floor "phantom chord."
- `piano_transcription_inference`: piano-specific model with pedal detection, but install/provenance was a genuine blocker (checkpoint auto-download via `wget` silently failed on macOS with no wget installed; manually staging the checkpoint required downloading and pickle-deserializing an external binary, which Claude Code's safety layer correctly declined to do autonomously without explicit user authorization). Once the user manually provisioned the checkpoint and it WAS tested: confidence/velocity showed **no separation** between real notes and phantoms (phantom velocities overlapped directly with real-note velocities), and it ran at ~1x realtime on CPU (15-28x slower than basic-pitch) — a real UX cost for no accuracy benefit.
- `basic-pitch`: also produces phantom octave-harmonic notes (this is NOT fully solved by any of the three approaches), but its confidence score showed real, exploitable separation between real notes (~0.5-0.85) and phantoms (~0.29-0.53) in validation testing, and it runs 15-30x faster than `piano_transcription_inference` on CPU. This confidence-separation property is the direct foundation for the group-relative filtering approach (next point).

**Confidence filtering is group-relative (per-event), not a single global threshold — because a global threshold was tried first and broke on real chord data.** Stage 2 of the basic-pitch migration implemented `CONFIDENCE_THRESHOLD = 0.6` as a single global cutoff. Testing against `chords-notes-mix.mp4` showed this **completely erased three entire 4-note chords** — because basic-pitch appears to split its confidence "attention" across simultaneously-sounding notes, so a real chord tone can legitimately score LOWER than a phantom note elsewhere in a monophonic passage. A global threshold cannot distinguish these cases. Stage 3 replaced it with `filter_event_notes`: group notes into events FIRST (chord-aware), then filter each note against its OWN event's strongest note (`RELATIVE_CONFIDENCE_FRACTION`) plus a low absolute floor (`ABSOLUTE_CONFIDENCE_FLOOR`) to still catch genuinely-isolated noise. This immediately rescued all three previously-erased chords in the same test clip.

**Notation duration comes from the RHYTHMIC gap between quantized onsets, NOT basic-pitch's acoustic offset times — deliberately.** `calculate_note_durations` computes a note's notated duration as `next_event's_quantized_beat_position - this_event's_quantized_beat_position` (or to end-of-piece for the last event). This is fundamentally different from `offset - onset` from basic-pitch's raw detection, which reflects acoustic decay (how long the string was actually still vibrating/audible) — a piano note's acoustic tail is often much longer than its notated rhythmic value (e.g. a quarter note played staccato-to-legato still decays audibly well past the next beat). Using acoustic offsets for notation would produce wildly overlong, wrong-looking note values. The rhythmic-gap approach is what actually answers "what note value did the performer intend," which is what notation needs.

**LilyPond/abjad chosen for rendering — PDF only, MusicXML deliberately stays a placeholder.** LilyPond's core strength is PDF/print engraving quality; it is not primarily a MusicXML-authoring tool, and abjad (its Python API) is built around constructing scores for LilyPond rendering, not MusicXML export. Rather than build a second, parallel, half-supported MusicXML-generation path, the explicit decision was: real PDF generation now, MusicXML stays a static placeholder until a dedicated MusicXML-generation effort is deliberately scoped (out of scope for everything done so far, including Stage C).

**Rhythm notation uses a full beat-respecting spelling algorithm (`spell_rhythm`) with double-dot support, not naive greedy duration decomposition.** An earlier version (Stage B) used a simple "greedily take the largest standard duration that fits, tie the remainder" approach. This produces musically illegal notation in many cases — e.g. a note starting off-beat with a 2-beat duration would render as one long tied blob instead of correctly splitting at each beat boundary it crosses, and dotted/double-dotted values weren't supported at all (so gaps that should be one dotted-eighth rest rendered as two separate tied rest symbols). Stage B2 replaced this with `spell_rhythm`: bar boundaries are the strongest constraint (never crossed), beat boundaries are respected except when a value legally starts on a beat (in which case it may extend past a subsequent beat boundary as a single dotted/double-dotted symbol — this is standard notation practice, verified against real notation convention), and only genuinely off-beat/partial spans get split at the beat level. **A documented, deliberate interpretation choice**: the algorithm widens rule 2's literal gate from "duration is a whole multiple of a beat" to "starts on a beat boundary" — the stricter literal reading would have incorrectly forced a beat-aligned double-dotted-quarter note to split into tied pieces instead of rendering as one notehead with two dots, which is objectively wrong per standard notation practice. This was explicitly flagged to the user as an interpretive judgment call, not silently decided.

**Other cases where an initial approach was implemented, tested, found insufficient, and replaced (each with a one-line reason — do not propose retrying these):**
- Fixed-length CQT chord-detection windows (one onset → one fixed-duration window) → replaced with onset-to-onset windowing, because fixed windows let one chord's analysis window bleed into the next chord's attack transient, contaminating pitch detection.
- `deduplicate_notes`'s first implementation only merged ADJACENT entries in one global onset-sorted list → replaced with per-pitch grouping first, because an unrelated note's onset sorting between two same-pitch fragments broke the adjacency chain and missed real merges (confirmed on real data: an E4 held across a chord change was never merged, appearing as a spurious extra "decay tail" event).
- The FIX for the above (per-pitch grouping with only a gap threshold, no span cap) → caused a NEW, worse regression: same-pitch notes from entirely separate musical events (different chords, seconds apart) chain-merged into one absurdly long fake note whenever consecutive raw fragments happened to have near-zero gaps. Fixed by adding `DEDUP_MAX_SPAN_SECONDS` (2.0s cap, justified by the longest genuine single-attack note actually observed in testing being ~1.65s). **This is the specific historical example referenced in Section 6** — the regression was caught only because the fix was re-validated against known-good ground truth data before being accepted, not just visually spot-checked.
- A proposed decay-tail-suppression rule (duration + pitch-recency) was validated against `test.wav.mp4`'s 15 known-good real notes BEFORE implementation, specifically because the earlier global-confidence-threshold mistake had already shown that "looks fine on one clip" isn't sufficient validation.

## 4. Known, Currently Unresolved Issues

Each item below was checked against the actual current code/most recent test results as of this writing — not assumed.

- **One decay-tail phantom note survives filtering.** Confirmed still present as of the most recent decay-tail-suppression testing: a note (F3 in the reference test clip) whose acoustic duration (~0.66s) clears the `DECAY_TAIL_DURATION_THRESHOLD` (0.4s) "is_short" gate, so the pitch-recency check never even runs against it. Not fixed — the threshold value was deliberately chosen based on real observed durations and this is a known, accepted gap at that threshold.
- **The "single C" ground-truth note in `chords-notes-mix.mp4` is never detected at any pipeline stage.** Confirmed still absent — this note never appears in `raw_notes` at all, meaning basic-pitch itself never detects it (not a filtering casualty). No pipeline-side fix is possible for this without changing the detection model or its parameters; this is a real accuracy ceiling of the current model on this specific passage.
- **One real chord tone (B4 in an E-G-B-D chord) is incorrectly filtered out.** Confirmed still happens — the relative-confidence filter drops it because an unusually strong co-sounding E4 in that same window drags the 60%-of-max relative bar high enough to exclude a real, but comparatively quieter, chord tone. This is an inherent tension in the group-relative filtering approach (Section 3) — a real note sharing a moment with a much louder note can still be cut. Not fixed.
- **Two phantom F3 leaks in specific chords** (an F-A-C-E-type chord and a D-F-A-C-type chord) survive relative-confidence filtering because their confidence, while lower than the chord's strongest note, doesn't fall low enough relative to it to trip the 60% bar. Confirmed still present in the most recent Stage C live-endpoint test.
- **Rhythm notation fragmentation from real timing imprecision against the quantization grid: STILL UNRESOLVED, no adaptive/multi-resolution quantization exists.** Confirmed by reading the actual current code: `quantize_beats` (line ~359 in `main.py`) still uses a single fixed 0.25-beat (sixteenth-note) resolution grid, unconditionally, for all input. There is no adaptive or multi-resolution quantization logic anywhere in the file. The most recent Stage C live-endpoint testing showed this produces heavily tied/dotted/fragmented-looking (though structurally correct and not garbled) notation when real onset timing doesn't land cleanly on the grid — flagged as a real readability concern and an explicit candidate for future work, but **no fix has been started, attempted, or validated**. Do not assume this has been addressed.
- **No tempo drift or time-signature-change support.** `tempo_bpm` is a single global scalar computed once per audio file via `librosa.beat.beat_track`; `NOTATION_BEATS_PER_BAR` and the hardcoded `(4, 4)` time signature passed to abjad are constants applied uniformly across the entire piece. A performance with a tempo change or a meter change partway through would be transcribed incorrectly (wrong bar placement past the change point). Not addressed at all.
- **The video/computer-vision layer has NOT been started.** The `extract_frame_at_time` function exists and is called in `analyze_audio`, but ONLY as a diagnostic sanity-check loop that prints whether a frame can be read at the first 3 note onset timestamps — it does **not** cross-reference detected notes against physically depressed keys in the video in any way. The entire "use the actual bird's-eye footage to validate/correct detected notes" concept from early multimodal-architecture planning is unimplemented. This is audio-only pitch/rhythm detection, full stop.
- **The music theory engine (chord naming, key detection) has NOT been built.** No key-signature detection, no chord-symbol/roman-numeral labeling, nothing. The rendered PDF shows raw detected pitches on a grand staff with a hardcoded key signature of C major (no sharps/flats key signature — accidentals are notated individually per note, not via a detected key signature).
- **`generate_placeholder_pdf` is STILL PRESENT in the code but is now genuinely dead code.** Confirmed by reading the file: the function (lines 34-70) is fully defined but its only call site was replaced by `generate_notation_pdf` during Stage C. It was deliberately left in rather than deleted (flagged explicitly in the Stage C session summary, not an oversight) since its removal wasn't explicitly requested. A new session should feel free to delete it if asked to clean up dead code, but should not assume its presence means placeholder PDFs are still being generated anywhere — they are not; `generate_notation_pdf` is what actually runs.
- **`generate_placeholder_musicxml` is still called and still produces a static, non-real placeholder.** This is intentional (Section 3), not a bug or oversight, and is explicitly out of scope until a dedicated MusicXML effort is planned.

## 6. Established Working Process

- **Repo health check before AND after any prompt that changes code**: `git status` (confirm clean/expected state), confirm `backend/venv/` is still untracked, confirm no stray files outside `backend/uploads/`, confirm any new Python dependency is declared in `backend/requirements.txt` (with its actual transitive dependency chain, not just the top-level package — e.g. when `abjad` was added, its full pulled-in dependency tree was captured too, not just `abjad==3.19` alone).
- **Test through the REAL live endpoint (`/api/transcribe`), not standalone scripts, for final validation.** Standalone scripts (always temporary, always deleted after use, never committed) are the right tool for isolated diagnosis of one specific mechanism — but before any pipeline change is considered validated, it must be exercised through an actual HTTP upload to the running server, checked against real response JSON, and (for anything visual/rendering-related) the actual generated file must be fetched and VIEWED as an image — not just confirmed to exist with a 200 status code.
- **Validate against existing known-good ground truth data BEFORE implementing a fix, and AGAIN after implementing, before declaring it fixed.** This is not optional process theater — it has caught real regressions. **Specific cautionary precedent**: the dedup span-cap regression (Section 3) — a fix for one confirmed bug (missed fragment merges) was implemented, and testing against the SAME real audio clip immediately revealed it had introduced a NEW, worse bug (same-pitch notes from unrelated chords chaining together into one fake multi-second note, causing real chord content to silently vanish from the API response). This was only caught because the "fixed" version was re-tested against the live endpoint with the same ground-truth clip before being accepted, not just assumed correct because the original bug was gone.
- **Prefer correct structural fixes over tunable-number patches when a problem is architectural, not cosmetic.** E.g., the global-confidence-threshold → group-relative-confidence-filtering change was a structural fix (Section 3), not a threshold retune, because the underlying problem (confidence isn't comparable across a whole piece) was architectural.
- **Git commits only happen when the user explicitly asks**, never automatically after every change. Multiple sessions have ended with validated, working changes sitting uncommitted in the working tree for the user to review before committing (sometimes across multiple sessions before a commit happened) — this is expected and fine, not a sign of an incomplete task.
- **When something looks like it might be a regression during testing, stop and diagnose the root cause rather than tuning around it** — several sessions have caught real bugs (not just parameter mistuning) this way, including the dedup span-cap issue above and an `abjad` pitch-language bug (Section 2/3) that only surfaced when real accidentals were exercised for the first time in Stage C, despite two full prior validation stages.
